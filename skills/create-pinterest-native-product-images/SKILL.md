# Create Pinterest Native Product Images

Create images that feel desirable, funny, strange, or saveable before they read
as promotion. Treat product truth and creative taste as separate hard
requirements.

## Load the right references

- Read `references/prompt-patterns.md` before writing generation prompts or
  reviewing a revision.
- When working in goose.gifts, also read `references/goose-gifts.md` before
  taking action. Follow the repository's current authorization and experiment
  state over any stale example in the skill.

## 1. Establish scope and authorization

Determine whether the request authorizes research, generation, editing,
shortlisting, pilot preparation, or public posting. Never infer that permission
to generate authorizes publishing, spend, subscriptions, accounts, or outreach.

If the task is planning-only, stop after concepts, prompt plans, and access
requirements. If generation is authorized, use the available image-generation
skill or tool and save project-bound outputs in the project.

## 2. Inspect existing work

Read the current marketing standard, experiment state, prior artifacts, reviews,
and next actions before proposing more images. Preserve rejected work and its
reason. Do not repeat an art direction merely because it is easy to prompt.

If a code knowledge graph exists, query it before broad code search when
locating creative-log tooling or publishing boundaries.

## 3. Verify the product before concepting

For every product-bound candidate:

1. Identify the exact listing and stable product ID.
2. Obtain the clearest available source image.
3. Inspect object type, silhouette, material, color, construction, markings,
   and likely scale.
4. Record uncertain facts explicitly.
5. Save the inspected reference in the project when reproducibility requires
   it.

Do not infer a different product from an ambiguous thumbnail. If identity or
scale cannot be verified, either keep the work as an unbound mood concept or
choose another product. Never call an image product-faithful without a source
reference.

## 4. Diverge before generating

Write at least three meaningfully different concepts before selecting one.
Change the visual mechanism, not just colors or camera angle. Favor:

- absurd editorial still life;
- beautiful room, one wrong thing;
- polished visual punchline;
- tactile field guide or specific-identity moodboard; and
- useful or aspirational scene interrupted by one ridiculous object.

State for each concept:

- intended audience or board;
- one-sentence hook;
- why someone might pause or save it without reading copy;
- product-fidelity risk; and
- what makes it unlike a reusable ad template.

Select concepts for a strong visual contradiction and a plausible native feed
context. Do not select merely because the product is easy to isolate.

## 5. Generate with separated references

Use the verified product image as the strict identity reference. Use mood or
composition images only as secondary references. Say which reference controls
which property.

Default to a vertical 2:3 image with:

- one immediately legible visual idea;
- believable scale and physical placement;
- intentional light, material, composition, and negative space;
- no logo, packaging, CTA, badge, price, border, or product-card frame; and
- no embedded text unless the concept specifically requires native editorial
  text and the user approved it.

Capture the exact prompt, tool, reference paths, artifact path, and relevant
settings. Do not reconstruct them later from memory.

## 6. Inspect and review

Visually inspect every generated artifact at full useful resolution. Score:

- scroll stop;
- Pinterest nativeness;
- aesthetic quality;
- concept clarity;
- saveworthiness;
- brand restraint;
- product fidelity; and
- AI artifact control.

Apply four hard gates:

- one clear idea;
- truthful product;
- no embedded CTA; and
- no ad-template structure.

Reject or revise any failed hard gate regardless of average score. Treat the
review as internal judgment until an owner reviews it.

## 7. Revise causally

Change only one or two diagnosed weaknesses per revision. Preserve the parent
attempt and state the change from parent. Useful revision targets include:

- object identity;
- scale;
- material;
- physical interaction;
- focal hierarchy;
- visual clutter;
- generic AI polish; and
- scene credibility.

If source verification disproves the premise, reject the old candidate and
create a corrected candidate. Do not silently relabel the failed execution.

## 8. Record the learning

When the project has a creative event log, append through its validator rather
than editing history manually. Record failed attempts, reviews, corrections,
decisions, and the concrete next action.

Save project-bound generated images and product references at durable,
repository-relative paths. Report the source reference, exact prompt, tool
mode, artifact path, score, gate result, and recommendation.

## 9. Prepare owner review

Show a small set of genuinely distinct survivors, normally two to four. For
each, explain:

- what earns the pause;
- what was verified against the product;
- remaining risk;
- where it would link; and
- whether it is concept-only, shortlisted, approved, or published.

Shortlisting never means publishing approval.

## 10. Learn from real distribution

Only compare public metrics with public cohorts. Keep Sandbox, preview,
internal, and public results separate. Treat unavailable metrics as unknown,
not zero.

After an authorized public test, record impressions, saves, Pin clicks,
site-side sessions, outbound clicks, checkpoint time, conclusion, and next
hypothesis.
